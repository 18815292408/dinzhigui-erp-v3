# 订单ID贯穿全流程 设计文档

## 问题

当前系统存在数据关联断裂问题：

1. **orders 表**使用 `customer_name`（文本）关联客户，而非 `customer_id`（UUID）
2. **designs 表**通过 `customer_id` 关联客户，但没有 `order_id` 字段
3. **installations 表**通过 `design_id` 关联设计，但也没有 `order_id` 字段

这导致：
- 设计方案无法直接读取关联订单的签单金额
- 订单号虽然唯一，但无法在设计方案中追踪
- 历史数据（如"大哥"的设计方案）因 customer_id 为 null 而无法关联订单

## 核心原则

**订单ID（order_id）作为唯一标识物，从始至终贯穿全流程。**

订单号（order_no）是系统内唯一不变的标识，用于串联所有业务环节。

## 业务流程

```
1. 导购创建订单（order_no 已生成）
         ↓
2. 导购派单给设计师
   - 订单状态 → pending_design
   - 发通知给设计师
         ↓
3. 设计师接单
   - 接单成功后才创建设计记录（draft）
   - designs.order_id = 该订单ID
         ↓
4. 设计师提交设计 → 安装管理（关联 order_id）
         ↓
5. 安装完成 → 订单完成
```

### 为什么接单后才创建设计记录？

设计师有可能不接单。如果派单时就创建设计记录，会导致：
- 设计师未接单但系统已有设计任务
- 数据状态不一致

接单后才创建设计记录确保：
- 设计师确认接受订单后才启动工作
- 流程更清晰严谨

## 数据库改动

### 1. designs 表增加 order_id 字段

```sql
ALTER TABLE designs ADD COLUMN order_id UUID REFERENCES orders(id);
CREATE INDEX idx_designs_order ON designs(order_id);
```

### 2. installations 表增加 order_id 字段

```sql
ALTER TABLE installations ADD COLUMN order_id UUID REFERENCES orders(id);
CREATE INDEX idx_installations_order ON installations(order_id);
```

## 现有数据修复

对于已有的设计方案（customer_id 为 null），需要通过订单号关联：

```sql
-- 找出设计方案对应的客户名，再匹配订单
UPDATE designs
SET customer_id = (
  SELECT c.id FROM customers c
  JOIN orders o ON o.customer_name = c.name
  WHERE o.customer_name = SPLIT_PART(designs.title, ' - ', 1)
  LIMIT 1
)
WHERE designs.customer_id IS NULL
  AND designs.title LIKE '% - 设计方案';
```

## 功能改动

### 1. 派单流程（dispatch）

**现有逻辑**：
```javascript
// 创建通知
await adminSupabase.from('notifications').insert({
  organization_id: order.organization_id,
  user_id: designer_id,
  type: 'new_order',
  priority: 'urgent',
  title: '新订单派发',
  summary: `客户 ${order.customer_name} 的订单已派给您，请及时接单`,
  related_order_id: orderId
})
```

**改动**：保持不变，只是更新订单状态和发送通知。

### 2. 接单流程（accept）

**现有逻辑**：
```javascript
// 只更新订单状态
const { data: order, error } = await supabase
  .from('orders')
  .update({ status: 'designing', ... })
  .eq('id', orderId)
  .select()
  .single()
```

**改动**：接单成功后自动创建设计记录

```javascript
// 接单成功后创建设计记录
await adminSupabase.from('designs').insert({
  organization_id: order.organization_id,
  order_id: orderId,
  customer_id: order.customer_id,  // 如果订单有 customer_id
  created_by: user.id,
  status: 'draft',
  title: `${order.customer_name} - 设计方案`,
})

// 更新订单状态
const { data: order, error } = await supabase
  .from('orders')
  .update({ status: 'designing', ... })
  .eq('id', orderId)
  .select()
  .single()
```

### 3. 创建设计页面（/designs/new）

**现有逻辑**：设计师可以选择任意客户创建设计

**改动**：
- 移除自由选择客户的功能
- 改为显示"分配给我的待设计订单"列表
- 设计师选择一个订单后，才创建设计记录
- 如果已有设计记录存在（订单已创建设计），直接跳转到设计编辑页面

### 4. 方案列表（/designs）

**现有逻辑**：显示所有设计方案

**改动**：显示订单号，方便追踪

```tsx
// 显示订单号
<div className="text-sm text-muted-foreground">
  订单号: {design.order_no || '无'}
</div>
```

### 5. 方案详情（/designs/[id]）

**现有逻辑**：
- 从 designs.final_price 读取成交价
- 从 designs.customers 关联客户

**改动**：
- 成交价从 orders.signed_amount 读取（通过 order_id 关联）
- 方案关联的订单信息从 order_id 读取

### 6. 安装管理（/installations）

**现有逻辑**：显示设计方案信息

**改动**：增加订单号显示，方便追踪完整流程

## 测试验证

1. **派单后设计师接单**：
   - 导购派单给设计师
   - 设计师收到通知
   - 设计师接单
   - 系统自动创建设计记录（draft 状态）
   - 设计方案能读取订单签单金额

2. **订单号追踪**：
   - 从方案列表能看到订单号
   - 从方案详情能看到关联的订单号
   - 从安装详情能看到关联的订单号

3. **历史数据修复**：
   - 运行修复 SQL
   - 已有设计方案能正确关联到订单
   - 成交价能正确显示

## 风险评估

1. **接单流程变更**：需要测试接单后创建设计记录的原子性
2. **已有数据**：需要迁移脚本修复历史数据
3. **前端改动**：创建设计页面逻辑较大调整

## 优先级

1. **高优先级**：数据库字段增加、接单逻辑改动
2. **中优先级**：界面显示订单号
3. **低优先级**：历史数据修复（可后续手动处理）
